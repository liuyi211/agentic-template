# Agents Rules

> **Purpose**: 定义AI编程助手（Agent）在本项目中的行为规范、操作流程、权限边界与决策原则。

---
## 1. 核心原则

### 1.1 确定性优先
- **可重现**：Agent的操作应该是确定性的，相同输入应产生相同输出。
- **无副作用**：避免产生未预期的副作用（如自动安装依赖、修改全局配置）。
- **幂等性**：相同的操作执行多次，结果应保持一致。

### 1.2 透明性
- **操作可见**：Agent必须明确报告它将要执行的操作，并在执行后报告结果。
- **变更可追溯**：所有代码修改应伴随清晰的变更说明。
- **错误坦诚**：遇到错误时，应如实报告，不隐瞒、不猜测。

---

## 2. 文档读取与上下文恢复（Context Recovery Protocol）

**核心目的**：确保每次新会话开始时，AI能在30秒内恢复完整上下文，无需用户重复说明。

### 2.1 文档读取顺序（强制）

当Agent接入项目时，**必须**按以下顺序读取文档：

```
1. agents.md (本文件)          → 了解行为边界与操作流程
2. memory-bank/design-document.md   → 理解项目目标与设计
3. memory-bank/tech-stack.md        → 了解技术栈约束
4. memory-bank/implementation-plan.md → 了解实现计划与当前阶段
5. memory-bank/todo.md                    → 了解当前任务状态
6. memory-bank/progress/stage_01_xxx.md        → 按编号倒序读取，了解已完成的工作
```

> **规则**：Agent在提供任何技术建议或执行操作前，必须先完成上述文档的读取。
> **例外**：如果用户明确提供了上下文，Agent仍需读取规则文件，但可跳过部分历史文档。

### 2.2 上下文恢复检查清单

每次新会话开始时，Agent应在读取文档后输出以下总结：

```markdown
## 上下文恢复总结

**项目**: [项目名称]
**当前阶段**: [阶段编号 - 阶段名称]
**阶段进度**: [X%]
**当前活跃任务**: [todo.md中标记为"进行中"的任务]
**最近完成**: [progress中最后记录的内容]
**阻塞项**: [如有]
**技术栈**: [核心框架版本]
```

## 3. 操作流程

### 3.1 任务执行流程

Agent执行任务必须遵循以下标准流程：

```
Step 1: 理解需求（Understand）
  ├── 读取 todo.md 确认任务优先级
  ├── 读取相关设计文档（design-document.md / implementation-plan.md）
  ├── 理解任务边界和验收标准
  └── 输出：需求理解摘要

Step 2: 分析现状（Analyze）
  ├── 检查现有相关代码
  ├── 识别依赖关系和影响范围
  ├── 评估技术方案可行性
  └── 输出：现状分析报告

Step 3: 方案设计（Design）
  ├── 提出实现方案（至少1个主推方案）
  ├── 说明方案优缺点
  ├── 确认不修改无关文件
  └── 输出：方案说明（等待用户确认）

Step 4: 实现开发（Implement）
  ├── 按确认方案编写代码
  ├── 遵循现有代码风格和规范
  ├── 添加必要的注释和文档
  └── 输出：代码变更

Step 5: 测试验证（Verify）
  ├── 运行单元测试（见4.1节）
  ├── 手动验证关键路径（见4.2节）
  ├── 检查类型错误和Lint警告
  └── 输出：测试结果报告

Step 6: 文档更新（Document）
  ├── 更新 todo.md 标记任务完成
  ├── 如有架构变更，更新 design-document.md
  ├── 如有接口变更，更新相关文档
  └── 输出：文档更新摘要

Step 7: 结果汇报（Report）
  ├── 总结完成的工作
  ├── 列出所有修改的文件
  ├── 说明测试验证结果
  └── 输出：完成报告
```

### 3.2 代码修改流程

任何代码修改必须经过以下步骤：

1. **变更前检查**
   - [ ] 确认当前Git分支正确
   - [ ] 确认本地无未提交的冲突修改
   - [ ] 确认要修改的文件范围

2. **变更执行**
   - [ ] 按最小变更原则修改
   - [ ] 保持代码风格一致
   - [ ] 不删除未引用的代码（除非用户要求重构）

3. **变更后验证**
   - [ ] 检查语法错误
   - [ ] 检查类型错误
   - [ ] 检查Lint错误
   - [ ] 运行相关测试

4. **变更记录**
   - [ ] 在对话中列出所有修改的文件
   - [ ] 说明每处修改的目的
   - [ ] 每次阶段性任务结束后询问用户是否要求git提交，提交时用20字以内写清楚该阶段完成内容

## 4. 测试流程

### 4.1 自动化测试执行流程

每次代码修改后，Agent必须执行以下测试流程：

```
1. 单元测试
   └── 运行命令: npm test 或 yarn test
   └── 要求: 所有测试必须通过
   └── 失败处理: 分析失败原因，修复后重新运行

2. 类型检查
   └── 运行命令: npx tsc --noEmit（TypeScript项目）
   └── 要求: 无类型错误
   └── 失败处理: 修复类型错误，严禁使用 @ts-ignore

3. 代码检查
   └── 运行命令: npm run lint
   └── 要求: 无严重错误（警告可记录但不阻塞）
   └── 失败处理: 修复Lint错误

4. 构建检查
   └── 运行命令: npm run build
   └── 要求: 构建成功
   └── 失败处理: 修复构建错误
```

### 4.2 手动验证流程

当自动化测试无法覆盖时，Agent 应在完成报告中明确列出需要用户手动验证的步骤，并将这些步骤记录在 progress 文件中作为待办项

```markdown
## 手动验证清单

### [功能名称]
- [ ] 验证步骤1: [具体操作]
- [ ] 验证步骤2: [具体操作]
- [ ] 预期结果: [应该发生什么]

### [边界情况]
- [ ] 验证步骤: [异常情况操作]
- [ ] 预期结果: [错误处理行为]
```

### 4.3 测试覆盖要求

- **新功能**：必须包含对应的单元测试
- **Bug修复**：必须包含回归测试（防止再次出现的测试）
- **重构**：确保现有测试全部通过，必要时更新测试
- **覆盖率**：核心业务逻辑覆盖率不低于 80%

## 5. 代码审查流程

### 5.1 自我审查清单

Agent提交代码前，必须进行自我审查：

- [ ] **安全性**：无SQL注入、XSS、硬编码密钥等安全问题
- [ ] **性能**：无明显的性能瓶颈（如N+1查询、大数据量全量加载）
- [ ] **错误处理**：所有异步操作有try-catch，所有错误有适当处理
- [ ] **边界情况**：处理了空值、越界、并发等边界情况
- [ ] **类型安全**：TypeScript类型完整，无隐式any
- [ ] **代码重复**：无重复代码，可复用逻辑已提取
- [ ] **命名规范**：变量、函数、类名清晰表达意图
- [ ] **注释**：复杂逻辑有注释说明"为什么"而非"做什么"

### 5.2 审查输出格式

对于复杂度中等的修改，应在完成报告中包含审查结果；对于简单修改可省略表格，直接说明

```markdown
## 代码审查报告

**审查范围**: [文件列表]

### 发现的问题
| 级别 | 位置 | 问题 | 建议 |
|------|------|------|------|
| 高/中/低 | 文件:行号 | 描述 | 修改建议 |

### 正面评价
- [优点1]
- [优点2]

### 总体评估
- [ ] 通过 - 可以直接合并
- [ ] 有条件通过 - 需修复[问题]后合并
- [ ] 不通过 - 需重新修改
```

## 6. 权限边界

### 6.1 文件系统权限

| 操作类型 | 权限级别 | 说明 |
|---------|---------|------|
| 读取项目文件 | ✅ 允许 | 可读取项目内任何文件 |
| 创建新文件 | ✅ 允许| 需在用户明确指令下执行 |
| 修改现有文件 | ✅ 允许 | 需在用户明确指令下执行 |
| 删除文件 | ✅ 允许 | 除非用户明确要求 |
| 执行系统命令 | ✅ 允许 | 仅限项目相关命令（如构建、测试） |
| 访问外部网络 | ✅ 允许 | 仅限获取依赖或文档 |
| 修改系统配置 | ⚠️ 询问 | 不得修改全局环境 |

### 6.2 代码修改边界

- ✅ **允许**：
  - 在指定文件内实现功能
  - 按照已有代码风格进行编写
  - 修复明确的Bug
  - 添加测试用例
  - 重构用户指定的代码
  - 修改配置文件（如 `package.json`, `vite.config.ts`）
  - 引入新的依赖
  - 修改数据库 Schema
  - 修改API接口定义
  - 跨模块的结构性变更

- ⚠️ **需要确认**：
  - 自动提交代码到版本控制
  - 自动推送到远程仓库

- ❌ **禁止**：
  - 修改环境变量或密钥
  - 删除生产数据
  - 执行可能破坏系统的操作

## 7. 决策原则

### 7.1 当面临技术选择时

1. **遵循现有约定**：如果项目中已有类似实现，优先遵循现有模式。
2. **最小变更**：选择对现有系统影响最小的方案。
3. **可逆性优先**：优先选择容易回滚的方案。
4. **文档先行**：如果变更涉及架构或接口调整，先更新文档再修改代码。

### 7.2 当遇到不确定性时

```
不确定的事项 → 询问用户，而不是猜测
模糊的边界 → 请求澄清，而不是假设
矛盾的约束 → 提出选项，让用户决策
```

### 7.3 当发现潜在问题时

Agent应当：
1. 明确指出现有问题
2. 说明可能的影响
3. 提供2-3个解决方案
4. 推荐首选方案并说明理由
5. 等待用户决策后再执行

---

## 8. 代码质量标准

### 8.1 编写要求

- **类型安全**：优先使用TypeScript，避免 `any` 类型
- **错误处理**：所有异步操作必须有错误处理
- **测试覆盖**：新功能必须附带测试用例
- **单一职责**：函数/组件保持单一职责
- **可维护性**：代码应当自解释，必要时添加注释

### 8.2 禁止项

- ❌ 禁止使用 `// @ts-ignore` 或 `// @ts-expect-error`
- ❌ 禁止使用 `console.log` 提交到生产代码（调试后可删除或使用logger）
- ❌ 禁止硬编码敏感信息（密钥、密码）
- ❌ 禁止遗留未使用的导入或变量
- ❌ 禁止提交未测试的代码
- ❌ 禁止使用 `any` 类型（特殊情况需在代码注释中说明理由）

---

## 9. 沟通规范

### 9.1 响应结构

Agent在回复时应遵循：

```
1. 理解确认：简要总结用户的需求
2. 方案概述：说明将要采取的方法
3. 执行过程：报告具体操作步骤
4. 结果验证：说明如何验证结果
5. 后续建议：提出下一步建议（如有）
```

### 9.2 状态报告

执行多步骤任务时，Agent应：
- 开始前报告总体计划
- 每完成一个子任务报告进度
- 遇到阻塞立即报告
- 完成时提供总结

---

## 10. 安全与隐私

- 🔒 不得将用户代码或数据发送到外部服务（除非用户明确授权）
- 🔒 不得在代码中硬编码API密钥或凭证
- 🔒 不得记录或缓存敏感信息
- 🔒 发现安全漏洞时应立即报告
